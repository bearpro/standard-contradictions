# MDL quickstart guide

## Что такое MDL

MDL — это специальный язык, разработанный для моделирования требований. В
отличие от других языков, используемых для управления требованиями, его главная
задача — формализовать саму сущность требования, а не просто передать её для
интерпретации внешнему инструменту. Задача, для которой проектировался этот
язык, — поиск противоречий между стандартами, то есть все требования из одного
стандарта сопоставляются с требованиями другого, и из модели самих требований
делается вывод о наличии противоречия. По этой причине этот язык — не язык
программирования: он больше похож на Prolog, переписанный с оглядкой на
синтаксис ML.  
Кроме синтаксиса и системы типов, MDL отличается формальной моделью, на которой
он построен. В рамках цели поиска противоречий между нормативными документами
обычная пропозиционная логика работает хуже, чем специализированная
**деонтическая логика** (логика норм), объединённая с **линейной темпоральной
логикой на конечных трассах** (LTLf). Всё вместе это называется "формализм
LTLf".

Единственное что можно сделать с MDL модулем MDL-модулем — это `solve`. Солвер
(обычно Z3) принимает на вход несколько модулей, объединяет их в одном общем
пространстве и пытается найти такую конфигурацию (расстановку значений в поля),
при которой одновременно все правила из всех модулей будут выполняться
совместно. В результате `solve` возвращает факт `sat`/`unsat` и дополнительную
информацию, объясняющую, как он пришёл к такому заключению.  
Для поиска противоречий отдельно важно использовать устойчивые имена, поэтому
часть этой инструкции будет посвящена выбору имён.

### Hello world

MDL не является языком программирования, поэтому написать Hello world на нём
невозможно. Но можно написать модель и набор правил, которые будут вычислять
приветственную строку:

```mdl
module hello_world

open std.strings

entity greeting: string
entity username: string

rule O greeting_contains_username:
    greeting = concat("Hello, ", username) always

fact username = "user"
```

Затем выполните

```bash
~ mdl solve ./mdl/examples/hello_world.mdl | \
  jq '.model.trace[0].entities["hello_world.greeting"]'
"Hello, user"
```

## Обзор концептов языка

Каждый файл исходного кода MDL является модулем со своим именем и должен
начинаться с объявления модуля:

```mdl
module my_module
```

Модуль состоит из конструкций верхнего уровня:

- Примитивные типы  
  Это типы, встроенные в язык

  | Тип       | Пример литерала   |
  | --------- | ----------------- |
  | `unit`    | `()`              |
  | `bool`    | `true`, `false`   |
  | `int`     | `3`               |
  | `decimal` | `1.5`             |
  | `rat`     | `3/7`             |
  | `string`  | `"Hello, world!"` |

  Из примитивных типов собираются пользовательские типы.

- Пользовательские типы, ключевое слово: `type`  
  Привычные для ML-языков алгебраические типы: record или discriminated union.

  Примеры:

  ```mdl
  # record
  type Probe = {
    grade: string,
    hardness: rat,
    corrosion_resistance: string
  }

  # union
  type Material =
    | StainlessSteel(Probe)
    | CarbonSteel(unit)
    | Titanium(unit)

  # record
  type Package = {
    is_sealed: bool
  }

  # record
  type Scalpel = {
    blade_material: Material,
    is_sterile: bool,
    package: Package
  }
  ```

  В стандартной библиотеке языка определены несколько полезных пользовательских
  типов, например:

  ```mdl
  # module std.collections
  type List<T> = Empty(unit) | Cons(T, List<T>)

  # module std.result
  type Result<T, TErr> = Ok(T) | Error(TErr)
  ```

- Сущности, ключевое слово: `entity`  
  Именованные и типизированные, они определяют те самые поля, расстановку
  значений в которых пытается сделать солвер. Иными словами, `entity` моделируют
  объект, к которому затем прилагаются требования.

  Пример:

  ```mdl
  entity scalpel: Scalpel
  entity package: Package
  ```

- Функции, ключевое слово: `func`  
  Функции нужны для того, чтобы сложные правила оставались читаемыми. Все
  функции строго чистые и обязаны возвращать значение. Функции не обращаются к
  сущностям напрямую, но могут получать их в составе аргументов при _вызове_ из
  правила.

  Примеры:

  ```mdl
  func add(a: int, b: int) -> int: a + b

  func can_be_used(scalpel: Scalpel) -> bool:
    scalpel.is_sterile
  ```

- Правила, ключевое слово: `rule`  
  Деонтические требования к сущностям. Каждое правило может ссылаться на все
  сущности, определённые в модуле (или импортированные из другого модуля), и
  использовать функции. Правило всегда имеет деонтическую модальность O/F/P, имя
  и тело. Тело правила состоит из LTLf-формулы, то есть значений типа `bool`,
  связанных темпоральными операторами. Сами `bool`-значения могут быть выражены
  любым способом (_вызовы_ функций, поля `entity`, любые выражения и константы),
  а темпоральные операторы доступны только в теле правил.

  Примеры:

  ```mdl
  rule F empty_password: password = "" always

  rule O use_sterile_scalpel: scalpel.is_sterile always

  rule O use_permitted_scalpel: can_be_used(scalpel) always
  ```

- Факты, ключевое слово: `fact`  
  Неименованные простые ограничения без деонтической модальности, действующие на
  всей протяжённости трассы. Они удобны для интеграции внешних средств или для
  быстрой проверки гипотез, но в финальном модуле их обычно быть не должно.

  Пример:

  ```mdl
  fact x = 3
  ```

- Для всех конструкций верхнего уровня доступны опциональные текстовые аннотации
  источника. Они необходимы для трассировки к источнику. Аннотации должны
  отсылать к элементам документа-первоисточника настолько детальным образом,
  насколько это возможно. Например, модуль чаще всего описывает некоторый
  документ целиком, а типы, правила и функции обычно соответствуют конкретным
  разделам или пунктам.

  Пример:

  ```mdl
  @ ГОСТ 32125-2013 Консервы мясные. Мясо тушеное. Технические условия
  module canned_meat

  @ Пункт 5.2.2
  type RawMeatMaterialProperties = {
      connective_tissue_proportion: rat,
      adipose_tissue_proportion: rat
  }

  @ Раздел 1
  type MeatType =
      | Beef(unit)
      | Pork(unit)
      | Mutton(unit)
      | Horse(unit)
      | Venison(unit)

  @ Раздел 1
  type CannedMeatGrade = First(unit) | Top(unit)

  @ Разделы 1, 5.2
  type RawMeatMaterial = {
      meat_type: MeatType,
      properties: RawMeatMaterialProperties,
      grade: CannedMeatGrade
  }

  @ Разделы 1, 5.2
  entity raw_meat_material: RawMeatMaterial

  @ Пункт 5.2.2
  rule O material_proportions_satisfy_bounds:
      let satisfies_bounds: bool =
          case (raw_meat_material.meat_type, raw_meat_material.grade):
              | (MeatType.Beef(), CannedMeatGrade.First()):
                  let non_muscle_tissue =
                      (raw_meat_material.properties.connective_tissue_proportion +
                       raw_meat_material.properties.adipose_tissue_proportion)
                  non_muscle_tissue <= 6/100
              | (MeatType.Pork(), CannedMeatGrade.Top()):
                  raw_meat_material.properties.adipose_tissue_proportion <= 30/100
              | _: true
      satisfies_bounds always
  ```

- Именованные значения, ключевое слово `let`  
  Как и `func`, `let` не меняет семантику модели, а служит только инструментом
  структурирования исходного кода. Синтаксис `let {name} = {value} in {body}`
  означает, что в теле `body` можно использовать `name`, имеющий значение
  `value`. Значение самого `let` является значением его `body`.  
  Ключевое слово `in` можно опустить в блочной форме, если за `value` следует
  перенос строки с отступом на том же уровне, что и исходный `let`.

  Примеры:

  ```mdl
  func add(a: int, b: int) -> int: let result = a + b in result

  func mul(a: int, b: int) -> int:
    let result = a * b
    result
  ```

- Сопоставление с образцом, ключевое слово `case`  
  Сопоставление с образцом в первую очередь необходимо для обработки вариантов
  объединений. Все ветки выражения `case` должны иметь один тип.

  Пример:

  ```mdl
  type NameOrNumber = Name(string) | Number(int)

  func is_valid(v: NameOrNumber) -> bool:
    case v:
        | NameOrNumber.Name(str): str != ""
        | NameOrNumber.Number(num): num != 0
  ```

- Временные операторы  
  Такие операторы имеют стандартную LTL-семантику. С точки зрения системы типов,
  применение временного оператора превращает выражение с типом `bool` в
  `temporal`. Временные операторы применимы только к выражениям с типами `bool`
  и `temporal`.
  - `... now` — выражение должно быть истинно в шаге, относительно которого
    оценивается правило, чтобы считалось соблюдённым
  - `... always` — выражение должно быть истинно на всех шагах, на которых
    применимо правило, для того чтобы правило считалось соблюдённым
  - `... next` — выражение должно быть истинно на следующем шаге после шага,
    относительно которого оценивается правило, чтобы правило считалось
    соблюдённым
  - `... eventually` — выражение должно быть истинно хотя бы на одном шаге,
    совпадающем или следующем за шагом, относительно которого оценивается
    правило
  - `... until ...` — выражение слева должно быть и оставаться истинным на всех
    шагах до шага, на котором истинным станет выражение справа, чтобы правило
    считалось соблюдённым. При этом выражение слева может быть непрерывно
    истинно _ноль раз_, то есть `false until p` эквивалентно `p now`
- Логические операторы `and`, `not`, `or`, `implies` применимы к выражениям с
  типом `bool` и `temporal`.

## Основные шаги подготовки MDL модели

Это рекомендуемый порядок написания MDL-модели для нормативных документов.

1. Внимательно изучить исходный документ  
   В ходе изучения важно обращать внимание на структуру описываемого в нём
   объекта и на требования, которые к этому объекту применяются.
2. Выявить все требования к объектам  
   Их можно пока записать в черновик или просто держать в голове.
3. Определить MDL-типы для будущих `entity`  
   Определить типы, описывающие модель данных таким образом, чтобы к ним можно
   было приложить предикаты, заключённые в требованиях. Типы должны быть близки
   к тому, как сам документ описывает свои сущности, но с фокусом именно на
   требования.
4. Записать все MDL `entity` с именами и типами  
   Не все типы обязаны быть `entity` напрямую: некоторые типы входят в состав
   типов `entity`. Также некоторые типы могут вовсе не быть частью `entity`, а
   использоваться только для промежуточных представлений данных в сложных
   правилах, но скорее всего вам это не понадобится.
5. Записать все требования в виде `rule`.  
   Требования должны обладать деонтической модальностью O, P или F, и временной
   семантикой. Если в требовании неочевидна временная структура, запишите его
   как `always`.
6. Если правила получились слишком длинными и сложными — вынесите часть логики
   из них в функции

   ```mdl
   module tmp

   open std.collections
   open std.strings

   type User = { name: string }

   entity user: User

   rule O not_empty_name:
     list.len(to_list(user.name)) > 0 always
   ```

   Если необходимо, части предикатов можно выносить в функции, например:

   ```mdl
   module my_site_spec.user

   # stdlib modules imported implicitly

   open std.collections
   open std.strings


   entity user: User

   func str_len(str: string) -> int:
       let str_list: List<string> = to_list(str)
       list.len(str_list)

   rule O not_empty_name: str_len(user.name) > 0 always
   ```

   По возможности следует избегать использования коллекций и рекурсий при
   определении правил, так как они вычисляются крайне тяжело.

## Выбор имён для объектов модели

Язык рассчитан на _межмодульную интероперабельность_, то есть ваш модуль должен
корректно обрабатываться вместе с другим модулем, который писал другой человек в
другое время, ничего не зная о вас (и вы тоже о нём не знаете). Это достигается
за счёт наличия рядом с языком механизма выравнивания моделей, который
анализирует типы, имена и структуру `entity`-объявлений, определённых в
нескольких модулях, и выдаёт наружу "alignment-модуль". В таком модуле
импортированы два (или более) исходных модуля и явно прописано равенство
определённых свойств моделей.

Для того чтобы такая система работала хорошо, необходимо минимизировать энтропию
ещё до того, как будет запущен алгоритм выравнивания моделей. Поэтому так важно
следовать следующим правилам о структуре и именовании объектов в модели.

1. **`entity` — это публичный объект модели** — объявляй через `entity` только
   предметные объекты, которые могут сопоставляться между модулями.

2. **Одна `entity` должна соответствовать одному устойчивому предметному
   понятию** — сущность должна обозначать самостоятельный объект, а не свойство,
   состояние или проверку.

3. **Не создавать `entity` для каждого требования** — требования выражаются
   через `rule`, а `entity` группирует требования вокруг предметного объекта.

4. **Имя `entity` должно быть существительным или именной группой** — имя должно
   отвечать на вопрос «что это?», а не описывать действие или обязанность.

5. **Использовать единственное число** — называй сущность как один типовой
   объект: `user`, `pipe`, `scalpel`.

6. **Имя `entity` должно быть стабильным, а не контекстным** — не включай в имя
   номер раздела, версию стандарта, источник или способ проверки.

7. **Не дублировать контекст модуля в имени `entity`** — если область уже задана
   модулем, не повторяй её в каждом имени.

8. **Использовать единый стиль записи имён** — `entity`, поля, функции и правила
   — `lower_snake_case`; типы и варианты sum-type — `PascalCase`.

9. **Имя типа должно описывать класс объекта, имя `entity` — конкретную сущность
   модели** — обычно `entity scalpel: Scalpel`; при нескольких объектах одного
   типа уточняй роль.

10. **Не использовать слишком общие имена** — избегай `object`, `item`, `data`,
    `thing`, если есть более точный доменный термин.

11. **Не использовать локальные сокращения** — пиши полные понятные имена, кроме
    устойчивых доменных сокращений вроде `id`, `url`, `gps`.

12. **Не смешивать языки в именах** — выбери один язык имён для модуля, обычно
    английский, и используй его последовательно.

13. **Сохранять терминологию исходного текста, но нормализовать форму** — бери
    термин из стандарта, но приводи его к устойчивой форме: единственное число,
    без артиклей и лишних слов.

14. **Не включать значения свойств в имя `entity`** — состояния вроде `sterile`,
    `approved`, `active` должны быть полями, если это не часть устойчивого
    термина.

15. **Структура `entity` должна отражать предметную структуру, а не структуру
    текста** — поля должны описывать объект, а не пункты, абзацы и номера
    требований.

16. **Выносить независимые предметные объекты в отдельные `entity`** — если
    объект имеет собственные требования или может выравниваться отдельно,
    объявляй его отдельно.

17. **Не выносить каждое поле в отдельную `entity`** — простые свойства должны
    оставаться полями типа, а не становиться сущностями.

18. **Использовать композицию для частей объекта** — части объекта описывай
    полями и вложенными типами; отдельной `entity` делай их только при
    самостоятельных требованиях.

19. **Связи между сущностями выражать через поля** — не прячь связи в имени;
    структура типа должна явно показывать отношение между объектами.

20. **Не кодировать модальность требования в имени `entity`** — обязательность,
    запрет и разрешение выражаются через `rule O`, `rule F`, `rule P`.

21. **Имена правил должны ссылаться на `entity`, но не заменять её** — правило
    должно быть читаемым и указывать объект, но объектом выравнивания остаётся
    `entity`.

22. **Имена полей должны быть предметными и локальными к типу** — внутри типа не
    повторяй имя самого типа без необходимости.

23. **Для булевых полей использовать форму предиката** — булево поле должно
    читаться как утверждение: `is_sterile`, `has_label`,
    `contains_instructions`.

24. **Для числовых полей указывать измеряемую величину и единицу измерения** —
    поле должно называться по смыслу измерения: `length_km`, `radius_mm`,
    `weight_kg`.

25. **При конфликте между краткостью и ясностью выбирать ясность** — имя должно
    быть достаточно коротким, но однозначным для другого автора.

26. **Использовать уточнения только для различения сущностей** — добавляй
    `inlet`, `outlet`, `primary`, `backup` и похожие уточнения только когда они
    реально различают объекты.

27. **Избегать отрицательных имён** — не называй сущность через отсутствие
    свойства; отрицание выражай полем и правилом.

28. **Не использовать имена, зависящие от реализации модели** — избегай
    `record`, `tuple`, `state_machine`, `data`, если это не предметный термин.

29. **Разделять родовое понятие и специализированное понятие** — если стандарт
    говорит и об общем классе, и о частных видах, моделируй оба уровня явно.

30. **Общие требования лучше привязывать к общей `entity`** — требование ко
    всему классу объектов должно быть записано через общую сущность, а не
    продублировано в частных.

31. **Предпочитать явные структуры неявным строкам** — ограниченные значения и
    структурированные данные лучше выражать типами, а не `string`.

32. **Не прятать несколько сущностей в одном поле** — если поле фактически
    содержит разные предметные объекты, раскрой их структурно.

33. **Не создавать искусственную «главную» `entity`, если в тексте её нет** — не
    добавляй `standard_model`, `system`, `document`, если к ним нет предметных
    требований.

34. **Если сущность есть в другом стандарте, выбирать наиболее общепринятое
    имя** — предпочитай термин, который с большей вероятностью совпадёт с
    другими модулями.

35. **Не использовать одинаковые имена для разных смыслов** — один термин в
    модуле должен обозначать одно понятие; неоднозначные слова уточняй.

36. **Согласовывать имя `entity`, имя типа и поля, которые на неё ссылаются** —
    используй один словарь для сущности, её типа и связанных полей.

37. **Поля должны группировать свойства вокруг естественного владельца** —
    свойство должно находиться в типе того объекта, которому оно предметно
    принадлежит.

38. **Не смешивать состояние объекта и событие** — состояние моделируй полем,
    событие — отдельной структурой или правилом только при необходимости.

39. **Проверять каждую новую `entity` тремя вопросами** — это самостоятельный
    объект? к нему могут быть требования в другом стандарте? его полезно
    выравнивать?

40. **Минимальный шаблон хорошей сущности** — хорошая сущность имеет предметное
    имя, тип со свойствами и правила, выражающие требования к ней.

41. **Признаки плохой `entity`** — плохой сигнал: имя про пункт документа,
    правило, модальность, состояние, реализацию, сокращение или слишком общий
    объект.

42. **Итоговое правило** — `entity` должна быть устойчивым публичным якорем
    модели, пригодным для межмодульного выравнивания.

## Типичные ошибки

- Использовать отрицание в именах запрещающих правил

  ```mdl
  # WRONG
  rule F not_stop: stop always

  # RIGHT
  rule F stop: stop always
  ```

- Добавлять _правила-заглушки_, которые ничего не проверяют, в духе
  `rule O user_is_human: true always`.  
  Такие правила сигнализируют о том, что ваша модель неполна. Если вы обнаружили
  у себя такое правило, вам следует либо пересмотреть модель и добавить в неё
  соответствующее поле, либо убрать правило совсем, если его действительно
  неразумно проверять в вашем контексте.

- Оставлять `fact` в финальной версии модуля, моделирующего требования Факты
  нужны для быстрой проверки модели, или для интеграции внешних инструментов,
  действительно имеющих данные о внешних _фактах_. Все требования должны
  моделироваться как `rule`, чтобы при конфликте было видно имя правила и
  источник.

- Строить модель от данных, а не от правил  
  MDL-модель должна быть ориентирована на проверку требований к объекту
  стандартизации, а не на моделирование самого стандарта. Например, у стандарта
  часто есть `author`, `published_date`, `reviewers` и т.п.  
  Если данные в исходном документе не связаны с требованиями — их не должно быть
  среди `entity`.

- Использовать большие let-привязки в правилах  
  Если ваше правило выглядит слишком сложным, лучше вынести часть его тела в
  функцию. Вы также сможете добавить аннотацию источника в функцию, чтобы
  улучшить трассируемость к источнику.

  ```mdl
  # WRONG
  @ Пункт 5.2.2
  rule O material_proportions_satisfy_bounds:
    let satisfies_bounds: bool =
        case (raw_meat_material.meat_type, raw_meat_material.grade):
            | (MeatType.Beef(), CannedMeatGrade.First()):
                let non_muscle_tissue =
                    (raw_meat_material.properties.connective_tissue_proportion +
                     raw_meat_material.properties.adipose_tissue_proportion)
                non_muscle_tissue <= 6/100
            | (MeatType.Pork(), CannedMeatGrade.Top()):
                raw_meat_material.properties.adipose_tissue_proportion <= 30/100
            | _: true
    satisfies_bounds always

  # RIGHT
  @ Пункт 5.2.2
  func satisfies_bounds(
    meat_type: MeatType,
    grade: CannedMeatGrade,
    proportions: RawMeatMaterialProperties) -> bool:
    case (meat_type, grade):
        | (MeatType.Beef(), CannedMeatGrade.First()):
            let non_muscle_tissue =
                (proportions.connective_tissue_proportion +
                 proportions.adipose_tissue_proportion)
            non_muscle_tissue <= 6/100
        | (MeatType.Pork(), CannedMeatGrade.Top()):
            proportions.adipose_tissue_proportion <= 30/100
        | _: true

  @ Пункт 5.2.2
  rule O material_proportions_satisfy_bounds:
      satisfies_bounds(
          raw_meat_material.meat_type,
          raw_meat_material.grade,
          raw_meat_material.properties) always
  ```

## Суперкомпактный справочник синтаксиса

```mdl
@ annotation text
module my.module

import "relative/path.mdl"
open std.collections

type Rec = { field: Type, flag: bool }
type Sum<T> = Case(value: T) | Empty(unit)
entity object: Rec

func f<T>(x: T, r: Rec) -> bool:
    let value = r.field
    true

fact boolean_or_temporal_expr
rule O rule_name when condition: condition always
strict rule F no_x: x always otherwise fallback always
override specific_rule > general_rule
```

- Типы: `unit`, `bool`, `int`, `rat`, `decimal`, `string`, `Type<T>`, `(A, B)`,
  `{ field_a: A, field_b: B }`, `type Union<A, B> = VariantA(A) | VariantB(B)`.
- Выражения: литералы, `name`, `qualified.name`, `f(x)`, `x.field`,
  `Type { f = v }`, `(a, b)`, `if c then a else b`, `let x = v in body`,
  `case x: | Pattern when guard: body`.
- Паттерны: `_`, литерал, имя, `(a, b)`, `{ field = p }`, `Type.Case(x)`,
  `Type.Empty()`.
- Операторы: `not`, unary `-`, `* / %`, `+ -`, `= != < <= > >=`, `until`, `and`,
  `or`, `implies`, затем postfix `always/eventually/next/now`.
- Правило:
  `[strict|defeasible|defeater] rule [O|P|F] name [when expr]: temporal_body [otherwise temporal_expr]`;
  по умолчанию сила — `defeasible`.
- Время: `p now`, `p always`, `p next`, `p eventually`, `p until q`.
